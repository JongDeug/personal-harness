## 개요

이 스킬은 **"git-flow로 feature를 열고, 그 안에서 TDD 사이클을 돈다"** 는 작업 방식을 명세한다.

TDD는 코드를 먼저 쓰고 테스트를 나중에 붙이는 게 아니다. **실패하는 테스트를 먼저 쓰고(Red), 그 테스트를 통과하는 최소 코드를 작성하고(Green), 통과를 유지한 채 정리하는(Refactor)** 순서를 한 기능 단위마다 반복한다.

핵심 원칙 한 줄: **테스트가 통과한 적 없는 코드를 추가하지 않는다.**

---

## 빠른 참조 (TL;DR)

```
1. git-flow 로 feature 브랜치 시작        (/feat {issue-key})
2. 작업을 작은 단위로 쪼갠다              (한 사이클 = 한 동작)
3. 각 단위마다 사이클 반복:
   🔴 RED     실패하는 테스트를 먼저 쓴다 → 실행해서 "실패"를 눈으로 확인
   🟢 GREEN   테스트를 통과하는 최소 코드만 작성 → 실행해서 "통과" 확인
   🔵 REFACTOR 통과를 유지한 채 중복/네이밍/구조 정리 → 다시 실행해서 통과 확인
4. 단위가 끝날 때마다 의미 있는 커밋
5. 모든 단위 완료 후 git-flow 로 마무리    (/finish-feat 등)
```

---

## 전제: git-flow 위에서 돈다

이 스킬은 **독립 실행이 아니라 git-flow 스킬과 짝을 이룬다.**

- feature 브랜치 생성 / PR 생성 / 핫픽스 등 **브랜치·배포 플로우는 전부 git-flow 스킬에 위임한다.** 여기서 직접 브랜치를 만들거나 PR을 열지 않는다.
- 이 스킬이 책임지는 범위는 **브랜치가 열린 시점부터 작업이 끝나기까지, 그 안에서 코드를 어떤 순서로 쓰느냐** 뿐이다.

작업 시작 시 현재 어떤 브랜치인지 확인한다.
- 이미 `feat/*` 브랜치 위라면 → 바로 TDD 사이클로 진입한다.
- 아직 브랜치가 없다면 → 먼저 git-flow 스킬(`/feat {issue-key}`)로 feature 브랜치를 만든 뒤 진입한다.

---

## 단계 0: 작업 쪼개기

사이클을 돌기 전에, 기능을 **테스트 한 개로 검증 가능한 작은 동작 단위**로 나눈다.

- 좋은 단위: "빈 입력이면 에러를 반환한다", "유효한 토큰이면 사용자 ID를 돌려준다"
- 나쁜 단위: "인증 기능 전체" — 한 사이클에 담기엔 너무 크다.

각 단위는 독립적으로 Red → Green → Refactor 한 바퀴를 돌 수 있어야 한다. 사용자에게 나눈 단위 목록을 먼저 제시하고, 위에서부터 하나씩 진행한다.

---

## 단계 1: 🔴 RED — 실패하는 테스트 먼저

1. 지금 구현할 **한 동작**에 대한 테스트를 작성한다. 아직 구현 코드는 건드리지 않는다.
2. 테스트를 **실제로 실행해서 실패하는 것을 확인한다.** (컴파일 에러로 인한 실패도 일단 Red로 인정하되, 이상적으로는 "단언 실패(assertion failed)" 형태의 실패를 본다.)
3. **실패를 확인하지 않고 다음 단계로 넘어가지 않는다.** 실패를 보지 못하면 그 테스트가 무엇이든 통과시키는 가짜 테스트일 수 있다.

> 원칙: 테스트가 처음부터 통과한다면(green-on-write), 그 동작은 이미 구현돼 있거나 테스트가 잘못된 것이다. 둘 중 무엇인지 먼저 규명한다.

---

## 단계 2: 🟢 GREEN — 통과하는 최소 코드

1. 방금 쓴 테스트를 **통과시키는 데 필요한 최소한의 코드만** 작성한다.
2. 이 단계에서는 우아함보다 **통과**가 목표다. 하드코딩이라도 통과하면 일단 Green으로 간다 (다음 Refactor에서 정리).
3. 테스트를 실행해서 **통과를 확인한다.** 이전에 통과하던 다른 테스트가 깨지지 않았는지(회귀)도 함께 확인한다.
4. 아직 안 만든 다른 동작을 미리 구현하지 않는다. **지금 테스트가 요구하는 것만.**

---

## 단계 3: 🔵 REFACTOR — 통과를 유지한 채 정리

1. 테스트가 초록인 상태에서만 리팩터한다. 중복 제거, 네이밍 개선, 구조 정리.
2. 리팩터는 **동작을 바꾸지 않는다.** 새 기능을 여기서 추가하지 않는다 (그건 다음 Red의 몫).
3. 리팩터 후 **테스트를 다시 실행**해서 여전히 전부 통과하는지 확인한다. 깨지면 리팩터를 되돌리거나 고친다.
4. 정리할 게 없으면 이 단계는 건너뛰어도 된다.

---

## 단계 4: 커밋

- 한 동작 단위(한 사이클, 또는 의미 있게 묶이는 몇 사이클)가 끝나면 커밋한다.
- 커밋 메시지는 프로젝트 컨벤션을 따른다. TDD라고 해서 "red", "green" 을 커밋으로 남기지 않는다 — **동작 단위**로 묶어 `test:` + `feat:` 를 함께, 또는 의미 있는 한 덩어리로 커밋한다.
- 프로젝트에 별도 커밋/검증 규칙(CLAUDE.md, lint, 통합테스트 등)이 있으면 그 규칙을 우선한다.

---

## 단계 5: 마무리

- 모든 동작 단위의 사이클이 끝나면 git-flow 스킬로 PR 생성 등 마무리 플로우(`/finish-feat` 등)를 진행한다.
- PR을 열기 전, **전체 테스트 스위트를 한 번 더 돌려 전부 초록인지** 확인한다.

---

## 규칙 (반드시 지킨다)

1. **테스트를 코드보다 먼저 쓴다.** 구현부터 쓰고 싶은 충동이 들면 멈추고 테스트로 돌아간다.
2. **각 단계에서 테스트를 실제로 실행한다.** "통과할 것이다" 라고 추정하지 않고 실행 결과를 본다.
3. **Red 단계의 실패를 눈으로 확인한다.** 실패를 못 봤다면 그 테스트는 아직 신뢰할 수 없다.
4. **한 번에 한 동작.** 사이클 도중에 떠오른 다른 동작은 메모해두고, 현재 사이클을 끝낸 뒤 새 사이클로 처리한다.
5. **브랜치/PR 플로우는 git-flow 스킬에 위임한다.** 이 스킬은 그 안의 코드 작성 순서만 책임진다.

---

## 안티패턴 (하지 않는다)

- ❌ 구현을 먼저 다 짜놓고 테스트를 나중에 몰아서 붙이기 → TDD가 아니라 그냥 사후 테스트다.
- ❌ Red 실패를 확인하지 않고 바로 Green으로 넘어가기.
- ❌ 한 사이클에서 여러 동작을 동시에 구현하기.
- ❌ Refactor 단계에서 새 기능 추가하기.
- ❌ 테스트를 통과시키려고 테스트를 약화시키기 (단언을 지우거나 느슨하게).
