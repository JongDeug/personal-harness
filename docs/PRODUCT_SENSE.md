# Product Sense

기술적 의사결정을 할 때 제품 관점에서 판단하기 위한 가이드.

## 핵심 질문

기능을 만들거나 기술적 결정을 내리기 전에:

1. **사용자가 이걸 원하는가?** — 가정이 아닌 근거가 있는가
2. **가장 적은 노력으로 검증할 수 있는 방법은?** — MVP 사고
3. **이걸 안 만들면 어떻게 되는가?** — 기회비용 판단
4. **유지보수 비용 대비 가치가 있는가?** — 1인 조직의 리소스는 한정적

## 의사결정 프레임워크

### 기능 우선순위 (ICE)

| 요소 | 설명 | 점수 (1-10) |
|------|------|-------------|
| **I**mpact | 사용자에게 미치는 영향 | |
| **C**onfidence | 성공 확신도 | |
| **E**ase | 구현 용이성 | |

ICE Score = I × C × E → 높은 순서대로 실행

### Build vs Buy vs Skip

- **Build**: 핵심 차별화 요소이거나, 기존 솔루션이 맞지 않을 때
- **Buy/Use**: SaaS, 오픈소스로 충분할 때 (인증, 결제, 분석 등)
- **Skip**: 지금 당장 필요하지 않을 때. "나중에"는 보통 "안 함"과 같다

## 사용자 피드백

- 정량: 분석 도구 (GA, Mixpanel 등)로 행동 데이터 수집
- 정성: 직접 사용하며 느끼는 불편함 기록
- 1인 조직의 장점: 본인이 첫 번째 사용자
