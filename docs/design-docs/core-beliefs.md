# Core Beliefs

조직의 기술적 의사결정을 이끄는 핵심 신념.

## 1. 동작하는 소프트웨어가 최우선이다

- 완벽한 설계보다 빠르게 동작하는 코드를 만든다
- 단, "빠르게"가 "대충"을 의미하지는 않는다
- 기술 부채는 인식하고 기록하되, 당장 해결하지 않아도 된다

## 2. AI와의 협업을 전제로 설계한다

- 코드는 AI가 읽고 수정할 수 있도록 명확하게 작성한다
- 문서화는 사람뿐 아니라 AI 컨텍스트로도 활용된다
- 반복적인 작업은 자동화하거나 AI에게 위임한다

## 3. 단순함이 확장성이다

- 1인 조직에서 복잡한 아키텍처는 유지보수 부채다
- 추상화는 3번 이상 반복될 때만 도입한다
- "나중에 필요할 수 있으니까"는 도입 이유가 아니다

## 4. 제품 관점으로 기술을 선택한다

- 기술 선택의 기준은 "사용자에게 더 빨리 가치를 전달할 수 있는가"
- 트렌디한 기술보다 검증된 기술을 선호한다
- 단, 생산성을 크게 높이는 새 도구는 적극 도입한다

## 5. 테스트는 자신감의 도구다

- 100% 커버리지가 아니라 핵심 경로의 안정성을 보장한다
- 통합 테스트 > 단위 테스트 (실제 동작에 가까운 테스트 선호)
- 테스트가 리팩토링을 방해하면 테스트를 고친다

## 6. 디자인 패턴은 해결책이 아니라 신호다

패턴은 암기해서 적용하는 레시피가 아니다. 코드에 내재된 문제와 힘(forces) — 변화의 방향, 의존성의 긴장, 요구사항의 상충 — 에 대한 반응으로 자연스럽게 드러나는 구조다.

- **Forces를 먼저 읽는다** — 패턴을 고르기 전에 "이 코드에 작용하는 변화의 압력이 무엇인가?"를 먼저 파악한다
  - 개별 정책이 독립적으로 변하는가(atomic variation) → Strategy 등이 자연스럽게 나타남
  - 여러 단계가 묶음으로 구조적 변화를 겪는가(structured variation) → Template Method 등이 자연스럽게 나타남
- **패턴을 강제하지 않는다** — 단순한 코드에 억지로 패턴을 씌우는 것은 과도한 설계다. 패턴 없이 충분히 명확하면 패턴을 쓰지 않는다
- **패턴을 버릴 줄 안다** — 요구사항이 변해서 기존 패턴이 맞지 않으면 미련 없이 걷어낸다. 패턴 위에 패턴을 겹쳐 쌓지 않는다
- **테스트 통과 ≠ 좋은 설계** — 테스트는 동작(behavior)을 검증하고, 패턴은 변화에 대응하는 구조(structure)를 결정한다. 둘은 별개의 관심사다
